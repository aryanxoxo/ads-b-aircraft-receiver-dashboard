1. Overview

This guide walks through the complete rebuild of a 1090 MHz ADS-B receiver station. The system passively monitors Mode S transponder broadcasts from commercial and general aviation aircraft, decodes position, altitude, speed, and callsign data, logs it locally, and feeds a live web dashboard.

The stack has four layers:

  • RF hardware — antenna, coax, optional LNA, RTL-SDR dongle
  • Decoder — dump1090-fa or readsb running on Raspberry Pi
  • Logger — Python script that polls dump1090 JSON and archives to CSV
  • Dashboard — static GitHub Pages site consuming the logged data
This guide assumes a clean Raspberry Pi 4 running Raspberry Pi OS Lite (64-bit). Commands are identical on a 64-bit Pi 3B+. A regular Linux laptop works too — skip the Pi-specific steps.

3. Raspberry Pi Setup

3.1 Flash the OS

Download the Raspberry Pi Imager from raspberrypi.com/software. Flash Raspberry Pi OS Lite (64-bit) to the microSD card. In the Imager advanced settings (gear icon), configure:

  • Hostname: adsb-station
  • Enable SSH with a strong password or public key
  • Set Wi-Fi SSID/password (or leave blank if using Ethernet)
  • Set locale and timezone to your location
OS Lite is preferred over Desktop — it uses less RAM and CPU, which dump1090 will appreciate.

3.2 First Boot

Insert the SD card, connect Ethernet, and power on. SSH in after ~60 seconds:

ssh pi@adsb-station.local

Update the system before installing anything:

sudo apt update && sudo apt full-upgrade -y
sudo reboot

4. RTL-SDR Driver Installation

4.1 Blacklist the DVB-T Driver

Linux ships with a generic DVB-T driver (dvb_usb_rtl28xxu) that claims the RTL-SDR dongle before our software can. Blacklist it permanently:

sudo nano /etc/modprobe.d/blacklist-rtl.conf

Add these three lines, then save and exit (Ctrl+X, Y, Enter):

blacklist dvb_usb_rtl28xxu blacklist rtl2832 blacklist rtl2830
Skipping the blacklist step is the #1 cause of "usb_open error -3" errors. Do this before plugging in the dongle.

4.2 Install librtlsdr

For the V3 dongle, apt is sufficient. For the V4 dongle, build from source (below):

sudo apt install -y rtl-sdr librtlsdr-dev

Building rtl-sdr-blog from source (RTL-SDR V4 dongle, recommended)

sudo apt install -y git cmake build-essential libusb-1.0-0-dev
git clone https://github.com/rtlsdrblog/rtl-sdr-blog.git
cd rtl-sdr-blog && mkdir build && cd build
cmake ../ -DINSTALL_UDEV_RULES=ON
make && sudo make install && sudo ldconfig
sudo cp ../rtl-sdr.rules /etc/udev/rules.d/
sudo udevadm control --reload-rules && sudo udevadm trigger

4.3 Test the Dongle

Plug in the RTL-SDR dongle, then run:

rtl_test -t

Expected output:

Found 1 device(s): 0: Realtek, RTL2838UHIDIR, SN: 00000001 Using device 0: Generic RTL2832U OEM Supported gain values (29): 0.0 3.7 7.7 ...
If you get "No supported devices found", re-check the blacklist and udev rules, then unplug and replug the dongle after a reboot.

5. Installing dump1090 / readsb

5.1 Install dump1090-fa

dump1090-fa (FlightAware fork) includes a built-in HTTP server that serves decoded aircraft data as JSON on port 8080 — this is what the Python logger polls.

sudo apt install -y dump1090-fa

If not in the apt repository for your OS version:

wget https://github.com/flightaware/dump1090/releases/latest/download/dump1090-fa_*.deb sudo dpkg -i dump1090-fa_*.deb

5.2 Configure dump1090-fa

sudo nano /etc/default/dump1090-fa

Key settings to review and set:

RECEIVER_TYPErtlsdr
RTLSDR_DEVICE0 (device index; 0 for a single dongle)
NET_OPTIONS--net --net-ro-port 30002 --net-sbs-port 30003
HTTP_PORT8080 (JSON API and map served here)
BEAST_OPTIONS--lat 37.42 --lon -122.18 (your GPS coordinates)
MAX_RANGE360 (km; physical limit for ADS-B at altitude)
Set your exact latitude and longitude. dump1090 uses this to calculate range statistics and draw the range ring on the map. Wrong coordinates produce wrong range data.

5.3 Enable and Start the Service

sudo systemctl enable dump1090-fa sudo systemctl start dump1090-fa sudo systemctl status dump1090-fa

Verify the JSON feed is live (give it 30 seconds after first start):

curl http://localhost:8080/data/aircraft.json | python3 -m json.tool | head -40

You should see a JSON object with an "aircraft" array. If the list is empty, wait ~1 minute — it depends on local air traffic.

5.4 Alternative: readsb

readsb is a drop-in replacement with better performance on Pi. Install and configure identically:

sudo apt install -y readsb sudo systemctl enable readsb && sudo systemctl start readsb

readsb serves the same JSON endpoint at the same port. The Python logger works with either.

6. Antenna Placement & RF Chain

6.1 Why Placement Matters

ADS-B is line-of-sight. Aircraft at 35,000 ft are visible up to ~400 km in theory, but terrain, buildings, and foliage block lower-altitude targets. The single most impactful upgrade you can make is moving the antenna higher and outside — not buying a better dongle.

6.2 Recommended RF Chain

From antenna to Pi, in order:

  1. Antenna (outside, as high as possible, clear sky view)
  2. Low-loss coax run, shortest practical length (every 3 m of RG-58 ≈ 1 dB loss at 1090 MHz)
  3. Inline band-pass filter (optional but recommended in urban RF environments)
  4. LNA / Pro Stick Plus (placed close to the antenna if possible, not the Pi end)
  5. RTL-SDR dongle plugged into Pi USB 2.0 port
Coax loss matters. If your run is longer than 5 m, use LMR-400 or LMR-195 instead of RG-58. Alternatively, move the Pi near the antenna and run Ethernet, not coax.

6.3 DIY 1090 MHz Dipole Antenna

If you do not have a commercial antenna, a simple vertical dipole takes 15 minutes to build:

  • Cut two pieces of stiff wire to exactly 69 mm each
  • Solder one to the centre pin of an SMA connector, one to the shield
  • Orient both elements vertically (collinear), one up and one down
  • Mount in free space away from metal surfaces and cable runs
This antenna has ~2 dBi gain and gives surprisingly good range for zero cost. The commercial FlightAware antenna has ~5 dBi with a ground plane, extending range by 20–30%.

7. Python Data Logger

7.1 Purpose

dump1090 holds only the last ~60 seconds of aircraft data in memory. The Python logger polls the JSON API every 5 seconds and appends new records to a CSV file for historical analysis and dashboard ingestion.

7.2 Install Dependencies

sudo apt install -y python3-pip python3-venv mkdir ~/adsb-logger && cd ~/adsb-logger python3 -m venv venv && source venv/bin/activate pip install requests pandas

7.3 Logger Script

Create ~/adsb-logger/logger.py and paste the following:

import requests, csv, time, os from datetime import datetime, timezone DUMP1090_URL = 'http://localhost:8080/data/aircraft.json' CSV_FILE = os.path.expanduser('~/adsb-logger/aircraft_log.csv') POLL_SECS = 5 FIELDS = ['timestamp','hex','flight','alt_baro','alt_geom', 'gs','track','lat','lon','vert_rate','rssi','messages'] def ensure_header(path): if not os.path.exists(path): with open(path, 'w', newline='') as f: csv.writer(f).writerow(FIELDS) def fetch_and_log(): try: data = requests.get(DUMP1090_URL, timeout=3).json() except Exception as e: print(f'[WARN] fetch failed: {e}') return ts = datetime.now(timezone.utc).isoformat() rows = [] for ac in data.get('aircraft', []): if 'lat' not in ac: continue # skip non-position messages rows.append([ ts, ac.get('hex',''), ac.get('flight','').strip(), ac.get('alt_baro',''), ac.get('alt_geom',''), ac.get('gs',''), ac.get('track',''), ac.get('lat',''), ac.get('lon',''), ac.get('vert_rate',''), ac.get('rssi',''), ac.get('messages',''), ]) with open(CSV_FILE, 'a', newline='') as f: csv.writer(f).writerows(rows) print(f'[{ts[:19]}] logged {len(rows)} aircraft') if __name__ == '__main__': ensure_header(CSV_FILE) print(f'Logging to {CSV_FILE} every {POLL_SECS}s ...') while True: fetch_and_log() time.sleep(POLL_SECS)

7.4 Run as a systemd Service

sudo nano /etc/systemd/system/adsb-logger.service

Paste the following, then save and exit:

[Unit] Description=ADS-B CSV Logger After=network.target dump1090-fa.service [Service] User=pi WorkingDirectory=/home/pi/adsb-logger ExecStart=/home/pi/adsb-logger/venv/bin/python logger.py Restart=always RestartSec=10 [Install] WantedBy=multi-user.target
sudo systemctl daemon-reload sudo systemctl enable adsb-logger sudo systemctl start adsb-logger sudo journalctl -u adsb-logger -f # watch live output

8. Dashboard

8.1 Architecture

The live dashboard (aryanxoxo.github.io/ads-b-aircraft-receiver-dashboard) is a static GitHub Pages site. Two operating modes:

  • Simulation mode — JavaScript-generated synthetic traffic (current default, safe for demos when hardware is offline)
  • Live mode — fetches the JSON API from the Raspberry Pi when reachable on the local network

8.2 Enabling Live Mode

Update the following constants in the main JS file in the repo:

const DUMP1090_URL = 'http://adsb-station.local:8080/data/aircraft.json'; const SIMULATION_MODE = false;
Browsers block mixed HTTP/HTTPS requests. Since GitHub Pages serves over HTTPS, you cannot fetch HTTP from the Pi directly in the public site. Solutions: (1) set up an nginx HTTPS proxy on the Pi, (2) serve the dashboard locally from the Pi itself, or (3) keep simulation mode for the public site and use a local HTTP copy for live data.

8.3 Serving the Dashboard Locally from the Pi

Simplest approach for a live local display:

cd ~/adsb-dashboard # your local clone of the repo python3 -m http.server 8000

Open http://adsb-station.local:8000 in any browser on the same network. No CORS issues, full live data.

9.2 Checking Signal Quality

In the dump1090 web map (port 8080), signal strength should show between -3 dBFS and -15 dBFS for nearby aircraft. Weaker than -20 dBFS suggests an antenna placement or coax issue.

Scan the 1090 MHz band to confirm antenna reception:

rtl_power -f 1089M:1091M:10k -g 30 -i 1 -1 scan_1090.csv

A peak above -50 dBm at 1090 MHz confirms antenna reception. A flat spectrum means the antenna is disconnected or the RF chain has a break.

9.3 Common Issues

usb_open error -3Permission issue → sudo usermod -aG plugdev pi then re-login
0 aircraft in JSONAntenna disconnected, wrong coax, or no air traffic overhead — wait 5 min
Excessive noise floorMissing band-pass filter, or USB 3.0 interference → use a USB 2.0 port
Logger not writing CSVCheck journalctl -u adsb-logger → usually a path permission issue
Dashboard shows demo onlySIMULATION_MODE still true, or Pi not reachable from browser host
Pi overheatingThermal throttle stops USB → add heatsink or small fan
